# 数栈实时发布台元数据设计

## 目标

Web demo 从 MySQL 读取页面元数据；demo 假数据也落在 MySQL 表中，不再从应用代码里回退 mock 数据。

## 连接信息

复制 `.env.example` 为 `.env`，填入实际连接：

```env
DATABASE_URL=mysql+pymysql://发布台元数据库用户名:密码@主机:端口/库名?charset=utf8mb4
DTSTACK_METADATA_DATABASE_URL=mysql+pymysql://数栈元数据库只读用户名:密码@主机:端口/streamapp?charset=utf8mb4
DATABASE_REQUIRED=false
GIT_WORKSPACE_ROOT=.cache/git-repos
GIT_OPERATION_TIMEOUT_SECONDS=20
GIT_AUTO_REFRESH_INTERVAL_SECONDS=600
GITLAB_BASE_URL=http://jnbygitlab.jnby.com
GITLAB_LOGIN_TIMEOUT_SECONDS=10
GITLAB_DEFAULT_PROJECT_URL=http://jnbygitlab.jnby.com/hangwenjie405085/independent_pj
```

- `DATABASE_REQUIRED=false`：保留配置项兼容历史启动方式；当前代码配置了 `DATABASE_URL` 后，查询失败会直接暴露错误。
- `DATABASE_REQUIRED=true`：未配置 MySQL 或 MySQL 不可用时报错，适合正式联调。
- `GIT_AUTO_REFRESH_INTERVAL_SECONDS=600`：后台每 10 分钟刷新已配置仓库提交记录；设置为 `0` 可关闭自动刷新。
- GitLab 用户名和密码在配置页维护，密码使用 `COOKIE_ENCRYPTION_KEY` 加密写入 `rc_gitlab_credential`，不会在 `.env` 中保存。
- `.env` 已加入 `.gitignore`，不要提交真实账号密码。

## 任务列表页当前约定

- `DATABASE_URL` 是 Web 发布台自己的元数据库连接，负责读取 `rc_*` 配置、发布记录和会话。
- `DTSTACK_METADATA_DATABASE_URL` 是数栈元数据库只读连接，任务列表页用它读取真实已提交任务。
- 数栈项目空间不是全局页面上下文；只有任务列表和发布确认展示“当前项目空间”，因为这两个页面需要确定发布来源空间与对应 Git 仓库。
- 首页是全局总览，不绑定单个项目空间；通过 `全部`、`实时`、`离线`、`项目空间` 标签切换统计视角。
- 首页右侧展示每日发布柱状图，按发布开始日期聚合最近日期的发布批次数。
- 配置管理是全局设置页，不读取顶部项目空间参数。
- 任务列表先选择测试项目空间，再从 `rc_git_repo` 读取该项目空间的 Git 仓库配置。
- 项目空间选择器在下拉项和当前值右侧展示模块小标，模块标识优先根据 `rc_datasource_resource.source_module` 推断为 `实时`、`离线` 或 `实时/离线`。
- 默认打开页面时不强制拉远端仓库；点击“刷新 Git 差异”后，会在 `GIT_WORKSPACE_ROOT` 下 clone/pull 当前项目空间绑定的跟踪分支。
- 服务端不使用 GitLab Web 登录 Cookie 执行 Git 操作；HTTP Git 认证使用配置页维护的 GitLab 凭据，并通过临时 `GIT_ASKPASS` 传递给 `git clone/fetch/pull`。
- 仓库第一次拉取时优先执行单分支 partial clone（`--filter=blob:none`），只下载提交与目录树等识别发布范围所需信息；如果内网 GitLab 不支持 partial clone，则自动回退到普通单分支 clone。
- 后端读取该项目空间最近一次成功发布的 `rc_release_batch.git_commit_id`，再用 `git diff 上次成功 Commit..当前 Commit` 生成变更文件列表。
- 任务列表页展示绑定仓库、跟踪分支、已发布 Commit、最新 Commit 和 Git 变更文件明细，用户可以先核对文件路径再进入发布确认。
- 目前解析 `.sql`、`.json`、`.yaml`、`.yml` 变更文件；根据路径中的 `realtime/stream/flink` 识别实时任务，根据 `offline/batch` 识别离线任务，任务名优先取文件名，`main.sql`、`release.yaml` 等通用文件名取父目录名。
- 离线 SQL 任务可以直接由 `.sql` 文件进入发布草稿；实时任务和离线数据同步任务必须由开发者手动提交 `manifest/export JSON/YAML`，否则任务列表会标记为 `缺少发布文件` 并在发布确认中阻断。
- 点击“刷新 Git 差异”后，系统会把本次扫描结果写入 `rc_release_draft`、`rc_task_artifact`、`rc_release_draft_task` 和 `rc_release_validation`；确认页通过 `draft_id` 读取草稿，不再只依赖 URL 中的任务 ID。
- 当前阶段发布确认只生成非 API 发布记录，写入 `rc_release_batch`、`rc_release_task` 和 `rc_release_step_log`，不会创建、更新或提交生产数栈任务。
- 任务类型列来自 Git 解析结果和数栈元数据：实时任务按 `streamapp.rdos_stream_task.task_type` 转换，离线任务按 `ide.rdos_batch_task.task_type` 转换。
- 后端再按任务名、任务模块、项目 ID 查询数栈元数据库，实时任务取 `streamapp.rdos_stream_task.submit_state = 1`，离线任务取 `ide.rdos_batch_task.submit_status = 1`，并关联 `ide.rdos_project` 与 `dt_pub_service.uic_user` 补全项目空间和提交人。
- 任务列表前端支持每列排序；分页默认每页 10 条，可切换为 10、20、30、40、50 条。
- Web 发布状态优先使用 `rc_release_task.source_task_id` 中已有记录；真实任务尚无发布记录时默认展示为 `未发布`。

## 配置管理页当前约定

- 源环境固定展示为 `测试环境`，目标环境固定展示为 `生产环境`。
- 配置管理页当前重排为 5 个主区块：元数据同步、GitLab 全局凭据、项目空间 Git 绑定、项目空间映射、数据源映射配置，统一保持白底卡片和表格型后台布局。
- 项目空间映射表头改为“测试环境 / 模块类型 / 生产环境 / 启用状态 / 匹配规则 / 最近同步 / 操作”，不再重复展示“源环境 / 目标环境”字样；环境语义默认固定为测试到生产。
- 项目空间 Git 绑定、项目空间映射、数据源映射三个表格都使用前端独立分页状态；默认每页 10 条，可切换 5 / 10 / 20 条，并保留至少 5 行高度避免卡片跳动。
- 项目空间映射和数据源映射都在卡片头部展示当前统计摘要，例如总数、已启用、待配置，以及离线/实时或资源状态分布，方便快速核对配置覆盖率。
- 项目空间映射以测试项目空间为主表，优先展示 `rc_project_mapping` 真实状态；如果数据库中暂无映射，则尝试按“同名 + 同模块”带出生产候选项目空间。
- 项目空间映射支持新增、编辑、启用/停用；保存后直接写入 `rc_project_mapping`，页面刷新只读取平台库中的真实状态，不再依赖浏览器本地缓存。
- 配置页当前的“手动同步元数据”已经收敛为轻量同步，只同步：
  - 测试/生产项目空间
  - 测试/生产全局数据源（`dt_pub_service.dsc_info`）
  - 项目映射候选
  - 数据源映射候选
- 不再在这个按钮里顺手全量刷新任务缓存和目录树，避免每次进入配置流程都扫描整个元数据库。
- 同步项目空间时先停用当前测试环境旧缓存，再启用本次从数栈元数据库读取到的未删除、非元数据项目空间。
- 普通页面刷新只读取 Web 元数据库中的 `rc_project_space` 与 `rc_datasource_resource`，不直接扫数栈元数据库资源表。
- 配置页的数据源映射改为环境级全局资源视角：测试和生产两侧都直接取 `dt_pub_service.dsc_info`，不再按项目空间拆分同名数据源，避免同一个生产数据源在多个项目空间下重复展示。
- 配置页作为全局配置面板，数据源候选读取 Web 元数据库中已同步的全局非元数据资源，后续保存映射时再绑定具体发布链路。
- 数据源同步过滤 `is_deleted = 0`、`is_meta = 0`，并排除名称包含 `meta` 或 `元数据` 的资源。
- 数据源映射配置页以测试环境真实数据源缓存 `rc_datasource_resource` 为主表，保证每个测试数据源都能展示出来；若存在 `rc_datasource_mapping` 记录，则自动回填生产目标数据源和映射状态。
- 数据源映射左侧测试数据源只读展示，右侧生产数据源通过下拉框人工选择；行操作包含“保存 / 启用或停用 / 清空”，卡片底部提供“保存配置”。
- 数据源“连通性”列不再做页面侧伪检测，直接展示同步自 `dt_pub_service.dsc_info.status` 的资源状态：`1 -> 正常`，`0 -> 失败`，未同步到状态时展示 `未知`。
- 数据源映射表的“启用状态”与“资源状态”分开展示：前者表示当前映射是否参与发布链路，后者表示源侧资源在数栈元数据库里的状态字段。
- 数据源映射保存不再依赖项目空间映射；当前口径是“测试全局数据源 -> 生产全局数据源”的一对一映射。
- 数据源新增映射时会在当前表格顶部插入空白编辑行，自动带出测试数据源类型，并阻止重复选择已存在映射的测试数据源。
- 数据源映射保存后直接写入 `rc_datasource_mapping`，再次进入页面时通过 `source_datasource_resource_id / target_datasource_resource_id` 回显，不再使用浏览器本地状态。
- `rc_datasource_mapping` 在 v2 DDL 中增加测试/生产数据源资源 ID 字段，后续发布时应优先按 ID 映射，展示名和文本替换字段只作为兼容展示。
- 目录层级以生产初始化同步到测试后的相同 `relative_path` 为准；发布确认会校验测试目录与生产目录相对路径是否一致，不一致时阻断。
- 如果页面没有数据，请先执行 `scripts/seed_demo_data.py` 将 demo 数据写入 MySQL。

## Git 入库与层级一致策略

- 第一版不自动写 Git。离线数据同步、实时任务等不能由数栈自动提交 Git 的任务，必须由开发者手动提交发布配置 artifact。
- Git 主分支仍是发布范围来源：只有主分支变更被识别并生成草稿后，才能进入发布确认。
- 生产目录先初始化同步到测试，测试和生产项目空间内保持同样相对目录；发布台保存目录树后用 `relative_path` 校验层级一致。
- 后续接数栈 API 时，只需要把当前“非 API 记录生成”替换为真实的创建目录、创建/更新任务、写依赖调度、提交运维步骤。

## GitLab 项目访问检测

配置管理页提供 GitLab 全局凭据维护和登录检测。当前只验证 GitLab Web 登录和指定项目页访问权限，不保存 GitLab Cookie。

当前链路：

1. 后端读取 `rc_gitlab_credential` 中的 GitLab 地址、用户名和密码密文，并用 `COOKIE_ENCRYPTION_KEY` 解密。
2. 使用同一个 `requests.Session()` 访问 `/users/sign_in`，解析页面里的 `authenticity_token`。
3. 提交 GitLab 登录表单，登录成功后继续访问 `GITLAB_DEFAULT_PROJECT_URL` 或页面输入的项目 URL。
4. 若 HTTP 状态为 `200`、最终 URL 没有回到 `/users/sign_in`，并且页面内容包含项目路径或项目名，则判定该账号可访问项目。
5. `404` 视为项目不存在或无权限，`403` 视为无项目访问权限，跳回登录页视为登录态失效。

第一版暂不使用 GitLab API Token。GitLab 项目到数栈项目空间的映射通过配置页维护在 `rc_git_repo`：每个测试项目空间绑定一个仓库地址和一个跟踪分支。

## Git 提交刷新闭环

- 配置页的“读取远端分支”使用 `git ls-remote --heads` 获取远端真实分支，凭据通过临时 `GIT_ASKPASS` 传递，密码不会出现在命令行参数和日志中。
- 用户保存的分支称为“跟踪分支”，含义是发布平台用于识别本项目空间变更文件的分支，不强行等同于 GitLab 项目的默认分支。
- 保存项目空间 Git 绑定时，后端会再次读取远端分支并校验跟踪分支必须存在；不存在的分支（例如仓库只有 `dev/master` 时保存 `main`）会被拒绝写入。
- 刷新提交前也会校验当前已保存的跟踪分支；如果历史配置里残留了不存在的分支，刷新会直接提示远端真实分支列表，不再进入 `git clone` 后才报错。
- “刷新提交”会 clone/pull 已绑定仓库的跟踪分支，读取最近提交并写入 `rc_git_commit`，同时更新 `rc_git_repo.latest_commit_id` 与刷新状态。
- “刷新全部 Git”和后台定时任务只处理已启用、仓库地址非空的当前绑定；定时刷新默认每 10 分钟执行一次。
- 任务列表使用当前项目空间绑定仓库的最新提交，与上次成功发布 Commit 比较，生成变更文件和待发布任务。


## 登录与 Cookie 会话

Web 登录使用生产数栈账号密码完成认证。浏览器端先使用生产登录公钥执行 `sm2.doEncrypt(password, publicKey, 0)` 并拼接 `04`，只提交 `password_ciphertext`，不会在 Form Data 中提交明文 `password`。前端 SM2 能力通过本地静态文件 `app/static/vendor/sm2.js` 提供，不依赖外部 CDN。登录成功后，生产环境 Cookie 会加密写入 `rc_auth_session`，浏览器只保存本系统 `rc_session`。

需要在 `.env` 中补充：

```env
COOKIE_ENCRYPTION_KEY=请使用 python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())" 生成
AUTH_SESSION_TTL_HOURS=8
AUTH_COOKIE_SECURE=false
AUTH_VALIDATION_INTERVAL_MINUTES=10
```

验证码由后端从生产数栈获取，登录页展示图片并由用户手工输入；后台 OCR 仅保留在 `demo/login.py` 调试使用。验证码请求产生的临时 Cookie 按验证码 `key` 在后端内存中短暂缓存，提交登录时复用，避免验证码和登录接口拆成两个独立会话。

当前阶段暂不调用 `verify_after_login` 类接口做二次校验；只要生产登录接口下发 `dt_token` 和 `DT_SESSION_ID`，就认为登录成功并加密落库。后续接入真实发布 API 时，再补充轻量登录态校验接口。
