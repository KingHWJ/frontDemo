# 1. 背景
+ 当前数栈存在测试环境与生产环境两套独立环境。开发人员主要在测试环境中完成实时/离线任务开发、调试和提交，生产环境需要手动创建或更新对应任务，配置依赖、数据源、环境参数、调度等信息，过程重复、易漏、不可追溯。
+ 本功能目标是建设一个独立的发布控制台，将测试环境中已确认可发布的任务，通过标准化流程发布到生产环境运维侧，同时结合 Git 管理发布范围，保证“哪些任务本次要发”可控、可追溯。

---

# 2. 核心目标
## 2.1 业务目标
1. 将测试环境中开发完成并提交的任务发布到生产环境。
2. 保留测试任务的核心代码、上下游依赖、任务配置、调度配置等信息。
3. 支持测试数据源到生产数据源的自动映射替换。
4. 支持通过 Git 主分支变更识别本次发布任务范围。
5. 提供发布前确认、发布执行、发布结果记录、失败日志查看能力。
6. 降低人工在生产数栈中重复创建任务、复制代码、配置依赖的成本。

## 2.2 技术目标
1. 后端通过 API 模拟数栈页面操作，不直接改生产库。
2. 平台页面负责参数选择、任务确认、配置检查、发布触发和结果展示。
3. Git 主要用于代码归档与发布范围控制，不完全承担任务配置发布职责。
4. 发布控制台维护`自身元数据库`，用于记录映射关系、发布记录、Git 提交记录、任务发布状态等。

## 2.3 当前第一版实现约束
1. 当前页面默认只读发布台自身元数据库快照，不在页面请求中直连测试/生产数栈元数据库。
2. 测试/生产元数据同步只通过两种方式触发：
   - 配置页手动点击“同步元数据”
   - 后台定时任务自动同步
3. 普通页面访问不会触发 Git clone、Git diff、数栈元数据库全量扫描。
4. 任务列表只有在点击“刷新 Git 差异”时，才会重新拉取 Git 并生成新的发布草稿。
5. 任务列表普通切换项目空间时优先读取该项目空间最近一次草稿缓存；若暂无草稿，则展示空态并提示手动刷新，不自动触发 Git 扫描。
6. 发布确认页、发布记录页、首页都优先读取平台库中的草稿、记录和缓存结果，避免重复扫描。

# 3. 当前达成的关键共识
| 序号 | 共识项 | 说明 |
| --- | --- | --- |
| 1 | 开发主场仍在数栈测试环境 | 当前团队习惯是在数栈中开发、调试、提交，v1 不强制改成 VSCode/Jenkins 流程。 |
| 2 | Git 参与发布范围控制 | 只有合并到 Git 主分支/master/main 的任务，才允许进入生产发布列表。 |
| 3 | 任务列表不再单纯从测试元数据库按“已提交状态”拉取 | 因为测试环境中可能存在已提交但暂不发布的任务，必须通过 Git 主分支差异确定本次发布范围。 |
| 4 | 一个项目空间对应一个 Git 仓库 | 不同项目空间任务名可能重复，因此必须按项目空间隔离 Git 仓库与发布范围。 |
| 5 | 发布动作由自建 Web 平台触发 | 类似 Jenkins 的参数化构建能力，但使用自建页面承载配置、确认和执行。 |
| 6 | v1 先处理页面可配置的任务配置 | 脚本中硬编码的 ES 地址、特殊脚本模式等复杂场景先列入后续增强。 |
| 7 | 资源参数/环境参数需要特殊处理 | 首次发布可从测试带入；已发布任务后续应优先读取生产当前配置，避免覆盖生产参数。 |
| 8 | 发布到生产后仍由运维/开发人员在生产侧执行任务 | 本平台负责“发布到生产运维”，不负责自动运行生产任务。 |
| 9 | 项目空间唯一识别不能只看名称或原始 ID | 同一环境下离线和实时可能出现同名、同 project_space_id 的项目空间，平台侧必须使用 `env_id + project_type + project_space_id` 作为唯一识别口径。 |

# 4. v1 功能范围
## 4.1 包含功能
| 模块 | 功能 | 说明 |
| --- | --- | --- |
| 登录 | 生产账号登录 | 登录时使用生产数栈账号，获取生产环境 Cookie/Session。 |
| 任务列表 | 按项目空间查看待发布任务 | 切换项目空间时拉取对应 Git 仓库主分支最新变更。 |
| Git 变更识别 | 根据上次发布 Commit 与当前主分支 Commit 对比 | 提取发生变化的文件，解析出任务名和任务路径。 |
| 发布确认 | 展示任务代码、依赖、数据源映射、环境参数、调度配置 | 支持发布前人工核对。 |

| 项目空间映射 | 测试项目空间映射到生产项目空间 | 原则上两边项目空间结构保持一致。 |
| 环境参数处理 | 首次发布取测试配置，二次发布优先取生产配置 | 防止后续发布覆盖生产资源参数。 |
| 任务发布 | 创建/更新生产任务 | 通过生产环境 API 完成目录、任务、依赖、参数、调度等发布操作。 |
| 发布记录 | 记录每次发布结果 | 包括发布人、时间、任务、Commit、状态、错误信息。 |
| 配置管理 | 管理项目空间、Git 仓库、数据源映射 | 支持后续新增项目空间和数据源。 |


## 4.2 暂不包含功能
| 功能 | 暂不处理原因 |
| --- | --- |
| 用户权限体系细分 | v1 先以可跑通为主，后续再细化个人账号、角色权限、租户权限。 |
| 脚本模式内硬编码地址替换 | 需要解析脚本文本，风险较高，v1 先不做自动替换。 |
| 自动执行生产任务 | 发布与执行分离，避免误触发生产任务。 |
| Git 分支合并操作 | v1 假设人工完成测试分支到主分支合并。 |
| 自动冲突检测 | Git 冲突由开发人员在合并阶段处理。 |
| Jenkins 集成 | 当前采用自建 Web 页面触发发布，Jenkins 可作为后续方案参考。 |

## 6.3 项目空间选择
#### 作用
用于确定当前发布范围对应哪个项目空间、哪个 Git 仓库、哪套测试/生产项目空间映射。

#### 设计原则
1. 一个项目空间对应一个 Git 仓库。
2. 切换项目空间时，自动刷新该项目空间的待发布任务。
3. 项目空间名称在测试环境和生产环境应尽量保持一致。
4. 如果测试/生产项目空间不一致，必须通过配置管理进行映射。

#### 字段建议
| 字段 | 说明 |
| --- | --- |
| 项目空间名称 | 用户可见名称 |
| 测试项目空间 ID | 测试数栈项目空间唯一标识 |
| 生产项目空间 ID | 生产数栈项目空间唯一标识 |
| Git 仓库地址 | 该项目空间对应代码仓库 |
| 发布分支 | 通常为 master/main |
| 上次发布 Commit | 用于差异对比 |
| 当前最新 Commit | Git 主分支最新 Commit |

## 6.4 待发布任务列表
#### demo
<!-- 这是一张图片，ocr 内容为： -->
![](https://cdn.nlark.com/yuque/0/2026/png/1088961/1778654069297-43e6acf9-156e-4d8f-bd3a-a39d2f8900da.png)

#### 作用
+ 展示本次可发布的任务。任务来源不是简单查询测试库“已提交状态”，而是来自 Git 主分支变更文件。
+ 页面首次进入，或手动执行扫描，可获取最新代码任务

#### 任务来源逻辑
当前第一版先跑通 Git 侧识别闭环，发布文件类型只识别 `.sql`：

1. 离线和实时项目空间都先按 SQL 任务处理，任务名取 SQL 文件名去掉 `.sql` 后缀。
2. 如果该项目空间没有任何成功发布记录，则当前跟踪分支下所有 `.sql` 文件都视为待发布任务。
3. 如果存在上次成功发布 Commit，则使用 `上次成功发布 Commit..当前最新 Commit` 的差异文件生成待发布任务。
4. `.json`、`.yaml`、`.jar` 等实时/Jar/manifest 发布文件类型后续再做配置化选择。
5. 当前阶段暂不按任务名查询数栈任务 ID，先用 Git 文件路径生成稳定占位 ID，后续接入数栈元数据/API 后替换。

## 6.5 发布确认页
#### demo
<!-- 这是一张图片，ocr 内容为： -->
![](https://cdn.nlark.com/yuque/0/2026/png/1088961/1778653639122-8d6c0b6b-64c3-454c-9e10-d30db6049187.png)

#### 作用
在真正调用生产 API 前，集中展示任务发布所需的关键信息，让用户核对并修正。

#### 页面结构建议
#### 1）基础信息
| 字段 | 说明 |
| --- | --- |
| 任务名称 | 当前任务名 |
| 任务类型 | 离线/实时/同步任务等 |
| 源环境 | 测试环境 |
| 目标环境 | 生产环境 |
| 项目空间映射 | 测试项目空间 -> 生产项目空间 |
| Git Commit | 当前发布对应 Commit |
| 文件路径 | Git 中对应文件路径 |


#### 2）代码信息
展示测试环境当前提交版本代码，可支持与 Git 主分支文件内容比对。

建议能力：

1. 展示测试元数据中的任务代码。
2. 展示 Git 主分支中的任务代码。
3. 标记两者是否一致。
4. 若不一致，给出风险提示。

#### 3）数据源映射
| 测试数据源 | 测试数据源 ID | 生产数据源 | 生产数据源 ID | 状态 |
| --- | --- | --- | --- | --- |
| 微信测试库 | 1001 | 微信生产库 | 9001 | 已配置 |
| 伯俊测试库 | 1002 | 微信生产库 | 9002 | 已配置 |


校验规则：

1. 任务中使用到的每一个测试数据源，都必须存在生产映射。
2. 若存在未映射数据源，阻断发布。
3. 映射只替换数栈数据源 ID，不处理脚本中硬编码地址。

#### 4）环境参数/资源参数
处理原则：

| 场景 | 参数来源 | 说明 |
| --- | --- | --- |
| 首次发布 | 测试环境参数 | 因生产尚无历史配置，只能先以测试参数为初始模板。 |
| 非首次发布 | 生产环境当前参数 | 避免用测试小资源配置覆盖生产已有配置。 |
| 用户手动修改 | 发布确认页修改后的参数 | 修改后随本次发布写入生产。 |


建议展示形式：

1. 参数以结构化表单展示。
2. 无法结构化解析时，以 JSON/文本块展示。
3. 支持手动编辑。
4. 修改后进入发布请求体。

#### 5）依赖关系
展示任务上下游依赖。

| 依赖类型 | 测试任务 | 测试任务 ID | 生产任务 | 生产任务 ID | 状态 |
| --- | --- | --- | --- | --- | --- |
| 上游 | ods_order_df | 123 | ods_order_df | 987 | 已匹配 |
| 下游 | dwd_order_detail | 124 | dwd_order_detail | 988 | 已匹配 |


校验规则：

1. 根据测试元数据依赖表读取依赖关系。
2. 发布到生产时，需要转换为生产环境任务 ID。
3. 若生产依赖任务不存在，可提示是否需要先发布依赖任务。
4. v1 可先阻断发布，并提示缺失依赖。

#### 6）调度配置
展示调度表达式、调度周期、补数据配置等。

处理原则：

5. 调度配置底层一般是字段或字符串，可从测试元数据读取。
6. 首次发布可带入测试调度配置。
7. 若生产已有任务，优先读取生产当前调度配置，避免覆盖人工调整。
8. 是否允许覆盖生产调度，应在发布确认页明确提示。

## 6.6 配置管理
#### demo
<!-- 这是一张图片，ocr 内容为： -->
![](https://cdn.nlark.com/yuque/0/2026/png/1088961/1778653553583-ed6b1d70-8e5f-4f85-85a8-a8467247c239.png)

#### 作用
维护发布过程中需要使用的基础映射关系。

#### 当前落地方式
1. 配置页中的测试环境和生产环境信息来自平台库 `rc_environment`，不再写死在前端模板中。
2. 项目空间、任务、目录、数据源通过双环境元数据同步写入平台库：
   - 测试环境：`DTSTACK_TEST_METADATA_DATABASE_URL`
   - 生产环境：`DTSTACK_PROD_METADATA_DATABASE_URL`
3. 项目空间映射按“同名 + 同模块类型”自动生成候选，写入 `rc_project_mapping`。
4. 数据源映射按“同名 + 同类型”自动生成候选，写入 `rc_datasource_mapping`。
5. 页面当前展示的是候选与已确认状态，后续再补充手工确认和维护接口。

### 6.6.1 项目空间配置
| 字段 | 说明 |
| --- | --- |
| 项目空间名称 | 展示名称 |
| 测试项目空间 ID | 测试环境 project ID |
| 生产项目空间 ID | 生产环境 project ID |
| Git 仓库地址 | 当前项目空间代码仓库 |
| 默认发布分支 | master/main |
| 是否启用 | 是否允许发布 |


#### 设计建议
1. 项目空间与 Git 仓库绑定。
2. 项目空间下任务名必须唯一，但不同项目空间之间可重复。
3. 待发布列表必须先选择项目空间后再展示。

### 6.6.2 数据源映射配置
页面建议分为两个区域：

4. 测试环境数据源
5. 生产环境数据源

并支持新增映射行。

3. 映射关系只保存数据源 ID 和展示名称。

| 字段 | 说明 |
| --- | --- |
| 项目空间 | 关联项目空间 |

| 本地仓库路径 | 服务端 clone 路径 |
| 发布分支 | master/main |
| 上次成功发布 Commit | 成功发布后更新 |
| 当前最新 Commit | 每次刷新时读取 |
| 认证方式 | SSH Key/Token |

#### 当前实现补充
1. GitLab 全局凭据保存在平台库，由发布台后端加密后读取，不再依赖前端写死配置。
2. 项目空间绑定仓库时，模块类型直接继承真实项目空间元数据，不允许页面手工切换离线/实时标记。
3. 跟踪分支必须来自远端真实分支列表，避免误填 `main/master` 造成刷新失败。
4. 仓库提交历史会落到 `rc_git_commit`，仓库最新状态会落到 `rc_git_repo`，供任务扫描和发布记录复用。

# 9. 校验规则
### 9.1 发布前必须校验
| 校验项 | 规则 | 不通过处理 |
| --- | --- | --- |
| Git 仓库 | 仓库可访问，主分支可拉取 | 阻断发布 |
| Commit 差异 | 能识别上次发布 Commit 到当前 Commit 的文件变更 | 阻断发布 |
| 任务存在性 | 变更文件能映射到测试数栈任务 | 阻断发布 |
| 项目空间映射 | 测试项目空间存在生产映射 | 阻断发布 |
| 数据源映射 | 任务使用的数据源均存在生产映射 | 阻断发布 |
| 依赖关系 | 依赖任务能映射到生产任务 | 缺失则提示先发布依赖 |
| 参数来源 | 首次发布可取测试，非首次发布优先取生产 | 异常时提示人工确认 |

### 9.2 风险提示
| 风险 | 提示 |
| --- | --- |
| Git 代码与测试元数据代码不一致 | 当前测试任务代码与 Git 主分支代码不一致，请确认是否已推送最新代码。 |
| 生产已有任务参数与测试参数不同 | 当前任务已发布过，本次默认保留生产参数。 |
| 数据源存在硬编码地址 | 当前版本只替换数据源 ID，不替换脚本中的硬编码地址。 |

# 18. V1 实际发布改造补充说明（2026-07）

## 18.1 主链路从“草稿”切到“待发布任务池”

当前实现已开始把主流程从“发布草稿”切换到“待发布任务池 + 真实发布批次”模式。

新的主链路：

1. 刷新 Git 差异
2. 识别变更 SQL 文件
3. 同步到 `rc_pending_release_task`
4. 任务列表直接读取 `rc_pending_release_task`
5. 用户选择一个或多个待发布任务进入确认页
6. `/confirm/publish` 创建一个真实发布批次
7. 后端执行生产离线发布 API
8. 回写 `rc_release_batch`、`rc_release_task`、`rc_release_step_log`
9. 成功后更新任务绑定和最近成功 Commit

说明：

- 旧 `rc_release_draft`、`rc_release_draft_task`、`rc_release_validation` 暂时保留，仅用于兼容历史入口。
- 新主流程不再把“草稿”作为唯一核心模型。

## 18.2 待发布任务池设计

新增主表：`rc_pending_release_task`

唯一识别口径：

- `source_project_space_id`
- `module_type`
- `git_path`

状态说明：

- `pending`：可发布
- `blocked`：存在阻断项
- `publishing`：正在发布
- `failed`：发布失败
- `success`：发布成功
- `unsupported`：当前模块暂不支持真实发布

Git 刷新规则：

1. 新变更任务：新增待发布记录
2. 已存在 `pending/blocked/failed`：更新最近 Commit、提交人、提交时间、校验信息
3. 已成功发布后再次变更：新增一条新的待发布记录，不覆盖成功历史

## 18.3 当前真实发布范围

当前 V1 只做离线 SQL 的真实发布。

### 支持

#### 新增任务

1. 读取 Git SQL
2. 找测试项目空间对应的生产项目空间
3. 检查生产目录
4. 目录不存在时逐层自动创建
5. 创建生产任务
6. 保存测试任务与生产任务绑定
7. 提交生产运维

#### 更新任务

1. 优先按 `rc_task_publish_binding` 查找生产任务
2. 绑定失效时按“生产项目空间 + 任务名”查同名任务
3. 命中多个同名任务则阻断
4. 命中单任务后读取生产详情
5. 用 Git 最新 SQL 更新代码字段
6. 保留生产已有资源、告警、调度等配置
7. 提交生产运维

### 暂不支持

- 实时任务真实发布
- 删除、改名任务
- 依赖同步
- 复杂数据源替换
- 自动回滚

## 18.4 当前阻断规则

### 阻断

- Git SQL 不存在或为空
- 项目空间没有生产映射
- 任务名为空
- 生产登录失效
- 生产存在多个完全同名任务

### 非阻断

- 数据源映射缺失
- 生产目录不存在
- 测试/生产目录暂不一致
- 生产不存在任务

提示口径：

- 生产目录不存在：`发布时自动创建`
- 生产不存在任务：`将创建生产任务`

## 18.5 新增代码模块职责

### `app/release_repository.py`

- 维护待发布任务池
- 创建发布批次
- 写入发布任务结果
- 写入步骤日志
- 维护任务绑定
- 读取项目空间映射、目录缓存和 Git SQL

### `app/dtstack_release_client.py`

- 封装生产离线 API
- 负责目录查询/创建
- 负责任务查询/创建/更新
- 负责提交生产运维

### `app/release_service.py`

- 编排一个批次下多个任务的发布执行
- 统一处理新增、更新、暂不支持、阻断、失败汇总
