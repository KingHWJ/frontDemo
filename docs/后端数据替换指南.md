## 1. 通用操作步骤

替换后端数据是指：把 `main.py` 某个路由中 `if settings.use_mock_data:` 分支删掉，使其走回原始的 `repositories.py` 查询路径。

### Step 1：配置数据库

在 `.env` 中填入数据库连接信息：

```bash
# 发布台自身元数据库（读写 rc_* 表）
DATABASE_URL=mysql+pymysql://用户名:密码@主机:3306/库名

# 测试数栈元数据库（只读）
DTSTACK_TEST_METADATA_DATABASE_URL=mysql+pymysql://用户名:密码@主机:3306/测试库名

# 生产数栈元数据库（只读）
DTSTACK_PROD_METADATA_DATABASE_URL=mysql+pymysql://用户名:密码@主机:3306/生产库名
```

### Step 2：验证数据库表存在

确认目标路由依赖的表已经在数据库中创建。参考 `sql/` 目录下的建表脚本：

- `sql/release_console_metadata.sql` — 基础表
- `sql/release_console_metadata_v2.sql` — 扩展表（GitLab 凭据、发布草稿、校验记录等）

### Step 3：关闭 mock 模式

```diff
- USE_MOCK_DATA=true
+ USE_MOCK_DATA=false
```

### Step 4：删除 mock 分支代码

以 `/home` 为例：

```diff
  @app.get("/home")
  def home(request: Request):
      home_view = request.query_params.get("view") or "all"
-     if settings.use_mock_data:
-         context = mock_base_context("home")
-         context["request"] = request
-         context.update(mock_home_context(home_view))
-         return templates.TemplateResponse(request, "home.html", context)
      overview = get_home_overview(home_view)
      ...
```

### Step 5：启动服务验证

```bash
cd /Users/hwj/Codes/MyProjects/frontDemo
uvicorn app.main:app --reload --port 8000
```

访问对应页面，确认正常渲染。如果报错，按以下顺序排查：

1. **403 / 登录页** → 鉴权中间件拦截了，检查 `.env` 中 AUTH 相关配置
2. **500 数据库连接失败** → 检查 DATABASE_URL 是否正确
3. **500 表不存在** → 执行建表 SQL
4. **Jinja2 UndefinedError** → 数据库返回字段与模板期望不一致

### Step 6：保留回退能力（可选）

如果想暂时保留两种模式的无缝切换（过渡期推荐），可以在路由中保留 `if settings.use_mock_data:` 判断，通过改 `.env` 来切换，直到所有路由验证通过后再统一清理。

## 2. 页面调用链详解

### 2.1 任务列表 `/tasks`

**路由文件：** [main.py](../../app/main.py#L437-L527)
**模板：** [task_list.html](../../app/templates/task_list.html)

#### 用户操作流程

```
进入 /tasks?project_space_id=1
    │
    ├── 切换项目空间 → 页面重新加载，读取该空间的最近草稿
    │
    └── 点击「刷新 Git 差异」→ 加 ?refresh=1，触发 Git 扫描
```

#### 路由逻辑（3 条分支）

```
/tasks?project_space_id=X
│
├─ 分支1：有 cached draft 且未点刷新
│   → selected_project_space()
│   → get_latest_release_draft_preview()    读 rc_release_draft + rc_release_draft_task
│   → get_source_project_spaces()           读 rc_project_space
│   → get_task_stats(tasks)                 纯统计（内存计算）
│   → 渲染 task_list.html
│
├─ 分支2：选了项目空间但无草稿 + 未点刷新
│   → selected_project_space()
│   → get_source_project_spaces()
│   → git_context = {status: "idle", ...}  提示用户点击刷新
│   → 渲染（空任务列表）
│
└─ 分支3：点击了「刷新 Git 差异」(?refresh=1)
    → get_git_diff_task_plan(project_space_id, refresh_git=True)
    → persist_release_draft(task_plan, username)
    → get_source_project_spaces()
    → get_task_stats(tasks)
    → 渲染
```

#### `get_git_diff_task_plan()` 调用链（刷新时触发）

```
get_git_diff_task_plan(project_space_id, refresh_git=True)
│
├─ get_source_project_spaces()
│   └─ SQL: rc_project_space JOIN rc_environment JOIN rc_git_repo
│      返回所有测试环境已启用的项目空间列表
│      涉及表: rc_project_space, rc_environment, rc_git_repo
│
├─ selected_project_space(project_space_id)
│   └─ 从上面列表中选出当前空间
│
├─ get_last_success_commit(project_space.id)
│   └─ SQL: rc_release_batch WHERE release_status='成功'
│      找到最近一次成功发布的 commit
│      涉及表: rc_release_batch
│
├─ git_repo_for_project(project_space)
│   └─ 构造 GitRepo 对象（纯内存）
│
├─ build_git_api_snapshot(repo, project_space, last_commit, refresh)
│   │
│   ├── 优先：GitLab API 方式（如果 rc_git_repo.api_sync_mode=1）
│   │   ├─ gitlab_api_service().list_commits()  → GitLab API HTTP
│   │   └─ gitlab_api_service().compare_commits() → GitLab API HTTP
│   │   需要: rc_git_repo 中有 gitlab_project_id + api_sync_mode=1
│   │         rc_gitlab_credential 中有正确的 GitLab 凭据
│   │
│   └── 回退：本地 Git 方式
│       └─ build_git_snapshot(repo, last_commit, refresh)
│           ├─ git clone/pull 到本地缓存目录
│           └─ git diff last_commit..HEAD 提取变更文件
│          需要: rc_git_repo.repo_url 可访问
│
├─ latest_commit_info_for_file() × N（变动的每个文件）
│   └─ git log -1 --format 获取文件最新提交信息
│
├─ build_changed_task_candidates(changed_files)
│   └─ 解析文件名，生成候选任务列表（纯内存）
│
└─ merge_git_candidates_with_metadata(candidates, project_space)
    ├─ get_task_metadata_rows(project_space_id, module_type)
    │   └─ SQL: rc_task_metadata WHERE project_space_id=?
    │      涉及表: rc_task_metadata
    ├─ get_dtstack_submitted_tasks()
    │   └─ SQL: 从测试数栈元数据库读取已提交任务
    │      涉及外部DB: streamapp.rdos_stream_task, ide.rdos_batch_task
    └─ 将 Git 变更信息 + 元数据 + 提交状态合并
```

#### `persist_release_draft()` 落库（刷新时触发）

```
persist_release_draft(task_plan, username)
│
├─ INSERT INTO rc_release_draft       → 草稿头
├─ INSERT INTO rc_task_artifact       → 任务文件缓存（每个任务一条）
├─ INSERT INTO rc_release_draft_task  → 草稿任务明细（每个任务一条）
├─ build_validation_items()           → 生成校验项
│   ├─ 元数据缓存校验
│   ├─ Git 文件可用性校验
│   ├─ 项目空间映射校验  → 读 rc_project_mapping
│   └─ 数据源映射校验    → 读 rc_datasource_mapping
└─ INSERT INTO rc_release_validation  → 校验明细
```

#### 涉及数据库表汇总

| 表名 | 读写 | 用途 |
|---|---|---|
| `rc_project_space` | 读 | 项目空间列表 |
| `rc_environment` | 读 | 环境定义（测试/生产） |
| `rc_git_repo` | 读 | Git 仓库绑定 |
| `rc_git_commit` | 读 | Git 提交缓存 |
| `rc_release_batch` | 读 | 最近成功发布的 commit |
| `rc_gitlab_credential` | 读 | GitLab 凭据（API 模式） |
| `rc_task_metadata` | 读 | 数栈元数据同步缓存 |
| `rc_project_mapping` | 读 | 项目空间映射 |
| `rc_datasource_mapping` | 读 | 数据源映射 |
| `rc_release_draft` | 写 | 发布草稿头 |
| `rc_release_draft_task` | 写 | 草稿任务明细 |
| `rc_task_artifact` | 写 | 任务文件缓存 |
| `rc_release_validation` | 写 | 校验明细 |
| 测试数栈元数据库 | 只读 | 已提交任务 + 项目信息 |

#### 切换到真实数据的条件

1. `.env` 中 `DATABASE_URL` 指向可用的自身元数据库
2. 自身元数据库中 `rc_*` 表已建好（执行 `sql/` 目录建表脚本）
3. `.env` 中 `DTSTACK_TEST_METADATA_DATABASE_URL` 指向可用的测试数栈元数据库
4. 至少有一个项目空间的 Git 仓库已配置且可访问
5. 项目中安装了 `git` 命令行工具（本地 Git 模式）
6. 或配置了 GitLab 凭据和 `api_sync_mode=1`（API 模式）
7. 删除 `main.py` 中 `/tasks` 路由的 `if settings.use_mock_data:` 分支

### 2.2 发布确认 `/confirm`

**路由文件：** [main.py](../../app/main.py#L533-L605)
**模板：** [confirm.html](../../app/templates/confirm.html)

#### 发布流程（4 步向导）

```
/confirm?step=1              选择待发布任务（从草稿中勾选）
/confirm?step=2              发布确认 + 校验 + 数据源映射配置
/confirm?step=3              发布中（进度展示，当前阶段无 API 调用）
/confirm?step=4              发布完成（展示结果摘要）

POST /confirm/publish        执行发布（生成发布记录）
```

#### Step 1：选择任务

```
/confirm?step=1&project_space_id=1
│
├─ selected_project_space(project_space_id)        读 rc_project_space
├─ 优先读 draft_id → get_release_draft_model()
│   └─ SQL: rc_release_draft WHERE id=?
│       + rc_release_draft_task WHERE draft_id=?
│       + rc_release_validation WHERE draft_id=?
│   涉及表: rc_release_draft, rc_release_draft_task, rc_release_validation
│
├─ 无 draft_id → get_latest_release_draft_model(project_space_id)
│   └─ SQL: rc_release_draft WHERE source_project_space_id=? ORDER BY created_at DESC LIMIT 1
│   └─ 然后调 get_release_draft_model() 读完整草稿
│
├─ confirm_selection(all_tasks, task_ids)
│   └─ 从草稿的任务列表中筛选用户选中的任务（默认选前3个）
│
└─ 渲染 step=1 视图
    展示：项目空间、任务列表（含勾选框）、Git 文件状态、目录状态
```

#### Step 2：发布确认

```
/confirm?step=2&project_space_id=1&draft_id=X&task_ids=1,2,3
│
├─ selected_project_space(project_space_id)
├─ get_release_draft_model(draft_id)               读草稿头+任务+校验
│   │
│   └─ validation_groups（按 task_id 分组）
│       每项校验包含: check_name, status(通过/失败/未检测), message, is_blocking
│       来自 rc_release_validation（任务列表刷 Git 差异时已生成并落库）
│
├─ get_datasource_mappings()                       读数据源映射
│   └─ SQL: rc_datasource_mapping
│       JOIN rc_datasource_resource ×2
│       JOIN rc_project_mapping
│       JOIN rc_project_space ×2
│   涉及表: rc_datasource_mapping, rc_datasource_resource,
│            rc_project_mapping, rc_project_space
│
├─ environment_labels()                            读环境名称
│   └─ SQL: rc_environment WHERE env_type='test'/'prod'
│   涉及表: rc_environment
│
├─ mapped_target_project_name(selected_space.id)   读目标项目名
│   └─ SQL: rc_project_mapping WHERE source=? AND is_enabled=1
│
└─ 渲染 step=2 视图
    展示：任务摘要、校验结果、数据源映射配置（只读）
```

#### Step 3 & Step 4：发布中/发布完成

不执行额外数据库查询，纯展示。Step 3 提示当前阶段不调用真实 API，Step 4 展示确认完成结果。

#### POST `/confirm/publish`：执行发布

```
POST /confirm/publish  (draft_id, task_ids...)
│
├─ get_release_draft_model(draft_id)               获取草稿数据
│
└─ simulate_release_from_draft(draft_id, task_ids, username)
    │
    ├─ summarize_validation_status(validations)     统计校验是否全部通过
    │
    ├─ INSERT INTO rc_release_batch                发布批次
    │   存储: batch_code, task_count, success_count,
    │         release_status, git_commit_id, publisher 等
    │
    ├─ INSERT INTO rc_release_task × 选中任务数     批次中每个任务的明细
    │   存储: task_name, task_type, submitter,
    │         sql_repo_path, release_status
    │
    ├─ INSERT INTO rc_release_step_log × 3          步骤日志
    │   三步: 发布前校验 / 元数据同步 / 生产任务发布
    │
    └─ INSERT OR UPDATE rc_task_publish_binding     任务发布绑定
        └─ 记录 source 任务在 target 项目空间的发布关系
        涉及表: rc_release_batch, rc_release_task,
                 rc_release_step_log, rc_task_publish_binding

重定向 → /records/{batch_id}
```

#### 涉及数据库表汇总

| 表名 | 读写 | 用途 |
|---|---|---|
| `rc_project_space` | 读 | 项目空间 |
| `rc_environment` | 读 | 环境定义 |
| `rc_release_draft` | 读 | 发布草稿头 |
| `rc_release_draft_task` | 读 | 草稿任务明细 |
| `rc_release_validation` | 读 | 校验明细 |
| `rc_datasource_mapping` | 读 | 数据源映射 |
| `rc_datasource_resource` | 读 | 数据源资源缓存 |
| `rc_project_mapping` | 读 | 项目空间映射 |
| `rc_release_batch` | 写 | 发布批次（publish 时） |
| `rc_release_task` | 写 | 发布任务明细（publish 时） |
| `rc_release_step_log` | 写 | 发布步骤日志（publish 时） |
| `rc_task_publish_binding` | 写 | 任务发布绑定（publish 时） |

#### 切换到真实数据的条件

1. 任务列表已切换到真实数据（草稿已在数据库中）
2. `DATABASE_URL` 可连接，所有 rc_* 表已创建
3. `.env` 中有正确的 `DTSTACK_TEST_METADATA_DATABASE_URL` / `DTSTACK_PROD_METADATA_DATABASE_URL`
4. `rc_environment` 表中有测试环境和生产环境的记录（env_type='test'/'prod'）
5. `rc_datasource_resource` 中有已同步的测试/生产数据源
6. `rc_project_mapping` 中有项目空间映射
7. 删除 `main.py` 中 `/confirm` 路由的 `if settings.use_mock_data:` 分支
